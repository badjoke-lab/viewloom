# PR-175 Kick candidate expansion plan

## Summary

This PR defines how ViewLoom should expand Kick candidate coverage before treating registry-backed collection as a product-quality surface.

## Why

The current Kick problem is not only that seed candidates are stored in code.

The candidate universe itself must improve through:

- curated imports;
- registry persistence;
- observed-slug feedback;
- missed-slug cooldown;
- future permitted discovery sources.

## What was added

- `docs/kick-candidate-expansion-plan.md`

## Key decisions

- The built-in seed list is a bootstrap source only.
- Operator-provided CSV/JSON/newline imports are the fastest zero-budget expansion path.
- Previously observed live slugs should be promoted in the registry.
- Repeatedly missed slugs should be cooled down instead of wasting attempts.
- Discovery sources require separate source review before implementation.
- Registry-backed selection is still not directory coverage.

## Scope

Docs only.

This PR does not:

- run a D1 migration;
- execute import SQL;
- add a discovery source;
- change collector runtime;
- claim Kick parity;
- claim global coverage;
- add paid services.
